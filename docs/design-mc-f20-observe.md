**Addendum to:** `docs/design-mission-control.md` §6.1 (Transport — `local.observed` endpoint kind).
**Date:** 2026-04-27.
**Status:** Decided. Adds the read-only registration path for terminal-launched CC sessions so they show up in the dashboard.

## Why this addendum

The spec's §6.1 enumerates `local.observed` as an endpoint kind: "external CC session (e.g., `cldyo-live`) observed via the hook stream. `write()` throws `NotControllable`." The schema supports it (`sessions.endpoint_kind` CHECK constraint includes `'local.observed'`), and the ingestor handles it correctly (events with a known `cc_session_id` are inserted regardless of kind). What's missing is the **registration path**: nothing today actually inserts a row into `sessions` with `endpoint_kind='local.observed'` for a `cldyo-live` invocation. Result: every JSONL file in `~/.claude/events/raw/` belongs to an unregistered `cc_session_id` and the ingestor drops the events.

F-20 closes the loop. After this lands, every `cldyo-live <repo>` invocation registers an observed session with MC v2 before exec'ing CC, and the resulting events flow into the dashboard. Read-only — the principal cannot type into a `cldyo-live` session from the dashboard (it owns its own TTY); but the dashboard sees what's happening, which agent is doing what, when work blocks, and the full audit log.

## Decision 1 — `cldyo-live` registers, then exec's claude with `--session-id`

The wrapper changes shape:

```bash
# Before (current cldyo-live, src/cli/cldyo-live:40)
exec claude --dangerously-skip-permissions --model opus "$@"

# After (F-20)
SESSION_UUID=$(uuidgen | tr '[:upper:]' '[:lower:]')
TASK_LABEL="${1:-${LIVE_ID}}"
# Build the body via `jq -n --arg` so JSON metacharacters in any of the
# upstream env vars (a `displayName` in bot.yaml with a quote, a TASK_LABEL
# with a backslash or newline) don't break the request or open a field-
# injection foot-gun. Per Echo's PR #54 review.
BODY=$(jq -n \
  --arg title "$TASK_LABEL (observed)" \
  --arg ccSessionId "$SESSION_UUID" \
  --arg agentId "$LIVE_ID" \
  --arg agentName "$LIVE_NAME" \
  --arg principalId "$PRINCIPAL_ID" \
  '{title:$title, kind:"local.observed", ccSessionId:$ccSessionId, agentId:$agentId, agentName:$agentName, principalId:$principalId}')
# Gate `--session-id` on registration success. If MC v2 is unreachable,
# fall back to letting CC mint its own UUID — events go under that UUID
# in raw/ and would match nothing once MC comes back up. Per Echo's
# PR #54 review (Q3-vs-Decision-1 contradiction): unconditional
# --session-id orphans events for the entire session if registration
# fails.
if curl -fsS -X POST http://localhost:8767/api/sessions \
     -H "content-type: application/json" -d "$BODY" >/dev/null; then
  exec claude --dangerously-skip-permissions --model opus \
       --session-id "$SESSION_UUID" "$@"
else
  echo "[cldyo-live] MC v2 not reachable; running unobserved" >&2
  exec claude --dangerously-skip-permissions --model opus "$@"
fi
```

The wrapper preserves the current `--dangerously-skip-permissions` flag from `src/cli/cldyo-live:40` — F-20 is purely about visibility, not about changing the principal's permission posture. The `--permission-mode auto` flip is **not** part of F-20; if and when it's wanted, that's its own decision.

Key shape choices:

- **UUID generated by the wrapper, not by MC v2.** This is the only way to bind the wrapper's pre-spawn registration to the post-spawn hook events without a chicken-and-egg lookup. CC's `--session-id <uuid>` flag (verified to exist via `claude --help`) accepts a caller-supplied UUID and tags every emitted event with it; EventLogger writes events into `~/.claude/events/raw/<uuid>.jsonl`; the ingestor matches on that uuid.
- **`--session-id` instead of `--resume`.** `--resume` would re-load a previous session; `--session-id` starts a fresh one with a known id. F-20 always starts fresh; reconnection-to-prior-`cldyo-live` is a follow-up (F-20.1).
- **Reachability is best-effort.** If MC v2 isn't running, the wrapper logs a warning and continues — the principal's terminal session must not break because the dashboard is down.

## Decision 2 — `POST /api/sessions` learns the `local.observed` shape

Today the handler unconditionally calls `spawnControlledSession` (`api/handlers.ts:574`). F-20 widens it to branch on `body.kind`:

- `kind: 'local.process.controlled'` (default if omitted) — existing path. Spawns CC subprocess.
- `kind: 'local.observed'` — new path. Validates `body.ccSessionId` (required, non-empty string, UUID-shape regex), inserts a row into `agents` (using the supplied `agentId`/`agentName` if provided, falling back to default), creates the task + assignment per the existing transaction, then **inserts a `sessions` row directly with `endpoint_kind='local.observed'` and `cc_session_id=body.ccSessionId`**, no `Bun.spawn` call.

State-machine transitions for an observed session:

- `queued → dispatched` fires immediately on POST success (the principal-launched terminal IS the dispatch).
- `dispatched → running` fires on the **first event ingested** for that session (a heuristic — the principal's first `bash` or `assistant.message`). The ingestor gains a small "if the session is `dispatched` and an event arrives, transition to `running`" hook. This keeps the working grid honest — observed sessions only show as `running` once they're actually doing work, not the moment the wrapper's curl finishes.
- `running → completed` **is auto-fired** when CC's `Stop` / `SessionEnd` hook lands in the event stream for that session. This was originally deferred (v1 plan was "leave as running, manual close via F-20.2") but Echo's PR-#54 review (2026-04-27) flagged the metrics consequence: stale `running` observed sessions silently inflate F-18 cycle time / wait time without bound. Reversed: fire the transition opportunistically on session-end hook events. **Trade-off accepted:** "principal finished" and "principal abandoned" are not distinguishable from the data; the recorded cycle time is "time the terminal session ran" rather than "time the principal considered the work complete". For F-18 metrics the recorded cycle time is the more useful signal regardless, and the principal can re-classify via F-20.2 if needed (now an audit-log correction rather than the only close path).

## Decision 3 — `write()` continues to throw `NotControllable`

The endpoint-resolver's `createObservedEndpoint` (`session/endpoint-resolver.ts`) already returns a handle whose `write()` throws `NotControllable`. The dashboard's drill-down `Input` affordance must hide or disable the textarea for observed sessions. Mirror the F-10 "ended session" gate — same UX, different rationale message:

> This is an observed terminal session. The principal's TTY owns input. Open the terminal to send messages.

## Decision 4 — Observed sessions render with a visual indicator

The working grid agent card and the focus area attention card both surface a small `observed` badge for `local.observed` sessions, distinguishing them from controlled ones. Same data the F-7 drill-down already gets via `endpoint_kind` on the session row; add it to the API projection in `db/working-agents.ts` and render a 10px pill.

This matters because the principal's interaction model differs: a controlled session is "click in, type, the agent responds"; an observed one is "click in, read what's happening; if you want to give input, switch back to the terminal." Surfacing the difference in the card prevents wasted clicks on the input box.

## Decision 5 — Cleanup is hook-driven, with principal override

The primary cleanup path is the auto-transition on `Stop` / `SessionEnd` hook events (Decision 2). Belt-and-braces follow-ups remain:

- F-20.2 — "Re-classify observed session" affordance in the drill-down. Lets the principal change a `completed` observed session to `cancelled` after the fact (the auto-transition can't tell finished from abandoned; the principal can).
- F-20.3 — Stale-session sweeper. Catches the edge case where the hook event was lost (filesystem issue, relay drop). PID-liveness check; transitions to `cancelled` with an audit-log note. Fallback only — the hook path covers the common case.

Both deferred. The hook-driven primary path is sufficient for v1 ship: F-18 metrics no longer skew because every observed session now has a bounded cycle time.

## Open questions

- **Q1 — Should observed-session events count toward F-18 metrics?** Yes — *now safe* because Decision 2 auto-fires `running → completed` on the session-end hook. Without that auto-transition, observed sessions would silently inflate cycle time / wait time without bound (Echo, PR #54 review). With it, every observed session has a bounded recorded cycle time and the metrics walker continues not caring about endpoint kind.

- **Q2 — Multiple terminals open at once on the same machine?** Each one calls `cldyo-live` independently and gets its own UUID, its own session row, its own assignment. Working grid shows N concurrent observed sessions under the principal's agent identity. No collision risk.

- **Q3 — `cldyo-live` failure mode if MC v2 is down?** Per the wrapper code in Decision 1, registration success gates whether `--session-id <wrapper-uuid>` is passed. If `curl` fails, the wrapper warns to stderr and exec's claude **without** the flag — CC mints its own UUID, EventLogger writes events under that CC-minted UUID, ingestor drops them. No registered session ever existed; no orphan rows, no half-state. Raw JSONL is still on disk; reboot MC v2 and *future* `cldyo-live` invocations participate. (Earlier draft of Q3 contradicted Decision 1 by claiming the unconditional `--session-id` would still orphan events for the entire session; that was wrong, fixed in cycle-2 per Echo's PR #54 review.)

## Deferred to F-20.1+

- F-20.1 — Resume an observed session on `cldyo-live` re-launch (instead of always starting fresh).
- F-20.2 — Manual "mark observed session complete" button in the drill-down.
- F-20.3 — Stale-observed-session sweeper based on PID liveness.
- F-20.4 — Backfill registration for already-running terminal sessions (would require principal UI to manually claim an in-flight `cc_session_id`).
