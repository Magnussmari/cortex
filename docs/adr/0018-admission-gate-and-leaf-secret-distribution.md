# ADR-0018 — Network-admission gate + leaf-secret distribution (sealed-to-pubkey, per-member, two authorities)

**Status:** accepted (ratified with Andreas 2026-06-27) · **Refines:** [ADR-0015](0015-two-tier-onboarding-and-admission-gate.md) (admission gate), [ADR-0013](0013-sovereign-federation-model.md) (sovereign federation — the leaf secret as transport PSK) · **Refs:** cortex#1142 (R1 — repurpose the O-4a flow as the admission gate), [ADR-0017](0017-surface-tooling-arc-bundles.md) (per-agent `allowedSkills`, the Pier gate), [ADR-0019](0019-federated-payload-encryption.md) (M3 payload encryption — resolves the cleartext residual; shares the seal primitive), `docs/design-admission-gate-leaf-secret.md` (the design this ratifies), `CONTEXT.md` §"Network-admission gate" + §"Joining a network"

## Context

ADR-0015 repurposed the registry's `register → PENDING → grant` flow as the **network-admission gate** (it gates roster membership, mints nothing) and named the **Pier** concierge. ADR-0013 established that the only artifact crossing the principal boundary on federation is the **leaf shared secret** — a transport PSK, not an identity credential — and recorded it as "exchanged out-of-band."

`design-admission-gate-leaf-secret.md` (C-1142 R1) surfaced that the admission gate is ~80% built (the state machine, admin-signed admit/reject/list routes, D1 + in-memory stores, migration `0007`, and the `cortex network admit` CLI all exist) but that **leaf-secret distribution is 100% net-new and undesigned**, and that two ratified documents are in tension: ADR-0013 says the secret is "out-of-band"; ADR-0015 / `CONTEXT.md` §188 say admission "hands you the leaf shared secret." Reconciling them changes the registry's posture (today it "signs nothing, holds no readable secret"). That was the load-bearing decision the design was HELD on. This ADR records the resolutions.

The public/private split is enforced by capability: the public onboarding channel is fronted by **Pier** (zero-authority, `discord-post` only — surfaces and notifies, **cannot admit**); the privileged admit + secret action runs only in a private channel by a trusted executor (Luna via FleetAdmit). This is the ADR-0017 `allowedSkills` boundary, not a code branch.

## Decision

**Q1 — Where the leaf secret lives in transit: sealed-to-pubkey (b′) primary, concierge-OOB (c) opt-in.**
The hub admin, at admit time, **seals the per-member PSK to the joiner's already-registered ed25519 pubkey** (ed25519→X25519 + libsodium `crypto_box_seal`). The registry carries only the **opaque ciphertext** — it never sees plaintext, so the "registry holds no readable secret" invariant survives (it holds a blob it cannot read, the way it already holds opaque pubkeys), and proof-of-possession is **intrinsic** (only the registered private key decrypts). **Concierge-OOB (c)** — the hub admin / Pier delivers the secret over a named secure channel, registry untouched — is retained as an **opt-in delivery mode** for the trusted bootstrap, surfaced as `cortex network secret add-member --deliver sealed|oob` (default `sealed`). This reconciles the ADR-0013 tension: the sealed-blob carrier is still **not** a cross-`nsc`-operator trust act (no identity credential transits), so ADR-0013's sovereignty point holds; the "out-of-band" consequence is amended to "out-of-band **or** sealed-opaque-through-the-registry."

**Q2 — Per-member PSK scope.** Each admitted stack gets its **own** PSK (its own hub `authorization` user), not one shared network-wide secret. Per-member is the only scope with **targeted revoke** (drop one member's user without re-keying everyone) and **leak attribution**. It costs a hub-side `authorization` reload per admit — automated by the admit agent (Pier/Luna). Per-network is tolerated only as a 2–3-stack manual bootstrap, never the target.

**Q3 — Admission is the source of truth for the roster.** The served network roster `members[]` is sourced from **ADMITTED** admission rows for that network, **not** derived from announced capabilities. Capabilities are an **orthogonal facet** — what an admitted member *offers*, layered on top of membership, never the thing that *confers* it. (This is the design's Gap-B, resolved to "source of truth" rather than the lighter "filter the derived view," because membership and capability are genuinely different concerns and conflating them is the bug class ADR-0015 warns against.)

**Q4 — Member proof-of-possession read endpoint.** A joiner learns they are admitted and fetches their sealed blob from a **member PoP-read endpoint** — a `GET` released to an ADMITTED member who signs the request with their **registered key** (no admin key needed; the signature *is* the authorization). The same endpoint serves admitted-status **and** the opaque sealed blob (b′ needs no extra auth — the blob is useless to anyone else). A **Pier notify** on the public channel ("you're in") is an **optional** convenience push, not the authority path.

**Q5 — Two separable authorities.** The decision keeps **two authorities distinct in code**:
- **registry-admin** — signs the admit decision (on `REGISTRY_ADMIN_PUBKEYS`); transitions PENDING → ADMITTED. **Mints no secret.**
- **hub-admin** — mints the per-member PSK, writes the hub `authorization` user + reloads the hub, and seals the bearer copy.

The admit route **never mints the secret**; secret minting is a separate hub-local act (`cortex network secret …`). For `metafactory` both authorities **collapse into Luna** (FleetAdmit Tier-2 runs admit **and** secret-delivery as one concierge action), but the code keeps them separable so a future network whose registry-admin ≠ hub-host still works.

**Q6 — Ship `revoke-member` + `rotate` in R1.** Revoke is meaningless without them. **`revoke-member` MUST cut transport** — it drops the member's hub `authorization` user and **reloads the hub**, not merely removing the roster row (a roster removal alone leaves a live leaf link). `rotate` re-mints + re-seals. A **scheduled rotation cadence** is deferred; on-demand rotate (suspected leak) + revoke (on `network leave` / eviction) ship now.

**Q7 — The cleartext residual is resolved by pulling M3 forward.** A leaked-but-revoked PSK still permitted **passive reads of cleartext-over-TLS `federated.>` payloads while it was live**. For the community / 3rd-party tier that residual is unacceptable, so **M3 payload encryption is pulled into this release** (→ [ADR-0019](0019-federated-payload-encryption.md)). M3 and this ADR's option b′ **share one `seal-to-principal` primitive** (ed25519→X25519 + `crypto_box_seal`): leaf-secret sealing uses the core directly; M3 layers an envelope-binding wrapper on top.

**Pier airgap (capability-enforced).** The **public** onboarding channel is fronted by **Pier**: zero bus authority, `discord-post` only — it surfaces requests and notifies, and **cannot admit or mint a secret**. The **private** channel is where a **privileged** bot (Luna) admits and delivers the secret. The boundary is the ADR-0017 `allowedSkills` grant (Luna → FleetAdmit; Pier → `discord-post`), default-deny `Skill` elsewhere — enforced by the grant, from one shared surface pack, not by an `if` in the handler.

## Consequences

- **Registry posture amended, invariant preserved.** The registry moves from "holds no secret" to "carries an opaque per-member ciphertext it cannot read." A one-line amendment to ADR-0013's "out-of-band" consequence records that the sealed-blob carrier is not a cross-`nsc`-operator trust act. Registry compromise leaks only ciphertexts (useless without member seeds).
- **The hub admit ↔ hub reload coupling is real and automated.** Per-member scope means each admit triggers a hub-local `authorization` add + reload. The admit agent (Pier/Luna under FleetAdmit Tier-2) performs both; net-new tooling is `cortex network secret {add-member,revoke-member,rotate}`.
- **Gap-A / Gap-B / Gap-C close.** Gap A: the request carries `network_id` (CLI flag + claim field + `upsertPending` signature + `(principal_id, peer_pubkey, network_id)` idempotency). Gap B: ADMITTED rows are the roster source of truth. Gap C: the member PoP-read endpoint (which also serves the sealed blob under b′).
- **Two authorities stay separable in code even where they collapse operationally.** Tests pin that the admit route mints nothing and that secret-minting requires the hub-admin path.
- **Build order** (design §8): Gap A → Gap B → secret tooling (verifier copy: hub `authorization` + reload) → delivery (b′ seal + ciphertext field + member PoP read, or the `--deliver oob` concierge runbook) → wire into Pier/FleetAdmit Tier-2. The shared `seal-to-principal` module (ADR-0019 L4) lands once and is shared with M3.
- **Reversibility note retained.** The `archive/model-a-hub-minted-creds` ref (ADR-0015) stays intact.

## Alternatives considered

- **(a) Plaintext secret in the admitted record / (b) plaintext registry-fetch endpoint.** Rejected: either makes the **registry hold a readable cross-party secret** — a honeypot that breaks the registry-holds-nothing invariant and means a registry compromise leaks every member's PSK for every network. b′ (sealed) is the strong variant that keeps the registry holding only opaque blobs.
- **Per-network PSK as the target scope.** Rejected as target: revoking one member forces a network-wide re-key, and a leak has no attribution. Tolerated only as a tiny manual bootstrap.
- **Derive the roster from announced capabilities (filter the derived view).** Rejected: conflates membership with what a member offers. Admission is the trust decision; capability is an orthogonal facet. Sourcing membership from ADMITTED rows keeps the two concerns honest.
- **Collapse the two authorities in code (admit route also mints the secret).** Rejected: convenient for `metafactory` (one admin-agent — Luna) but wrong for any network whose registry-admin ≠ hub-host, and it would put secret-minting behind the registry-admin gate. Keep them separable; let the *agent* (FleetAdmit) collapse them at runtime.
- **Keep the leaf secret purely out-of-band (ADR-0013 verbatim, no sealed path).** Rejected as the primary: it pushes the PSK through a non-E2E human channel (Discord DMs are not end-to-end encrypted) and makes rotation/revoke a manual runbook. Retained as the `--deliver oob` opt-in for the trusted bootstrap, where the simplicity is worth it and the channel discipline is acceptable.
