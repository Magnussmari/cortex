# ADR-0024 — Pluggable surface plugins: adapters and renderers, out-of-tree, arc-installable, compat-gated, in-process

**Status:** **accepted** (ratified by the principal 2026-07-11) · **Date:** 2026-07-09 (renderer fold-in + ratification 2026-07-11) · **Lifts the deferral in:** [ADR-0017](0017-surface-tooling-arc-bundles.md) §Alternatives L64–67 (0017 itself is NOT edited) · **Reverses:** the *"renderers are static across hot-reload"* decision documented at `src/renderers/types.ts:45-57` · **Refs:** epic [#1784](https://github.com/the-metafactory/cortex/issues/1784), S0 [#1785](https://github.com/the-metafactory/cortex/issues/1785), `docs/design-pluggable-adapters.md`, [ADR-0005](0005-mission-control-integration-architecture.md) §4 (demotes the dashboard ring buffer), [ADR-0012](0012-external-operator-account-isolation.md) (process-isolation precedent), [ADR-0018](0018-admission-gate-and-leaf-secret-distribution.md)/[ADR-0023](0023-federation-leaf-credential-model.md) (signing precedent), `src/gateway/gateway-adapters.ts`, `src/common/types/surfaces.ts`, `src/adapters/types.ts`, `src/renderers/{types,index}.ts`, `src/bus/surface-router.ts`, F-092 (config hot-reload), C-110 (web adapter), G-1111 §4.6 (fail-safe renderer pairing)

**Vocabulary (`CONTEXT.md` is authoritative).** A **platform adapter** ("adapter") is agent-bound, inbound *and* outbound, one live connection per surface. A **renderer** is non-agent-bound, outbound only, with a `start`/`stop` lifecycle. A **render target** is the surface-router's render-only subscriber contract — named `SurfaceAdapter` in code today, a misnomer, and **not an adapter**. A **surface plugin** ("plugin") is an adapter *or* a renderer packaged as a separately-installable unit; it declares its `kind`. A **bundle** is arc's delivery unit — a plugin ships *in* a bundle; they are not synonyms.

## Context

Cortex compiles all four surface adapters (`src/adapters/{discord,mattermost,slack,web}/` — 2 858 / 1 469 / 1 189 / 535 LOC) into core, and their platform npm deps sit in cortex's own `package.json:47-55`. Adding, upgrading, or removing a surface requires a whole-cortex release, and every stack carries every surface's dependency weight.

[ADR-0017](0017-surface-tooling-arc-bundles.md) extracted the surface *tooling* into per-surface arc bundles (`metafactory-discord`, epic #1171) and **explicitly deferred the adapter**: *"Extract the adapter too … Rejected for now — the adapter is deeply woven into the bus/dispatch/surface-router; a plugin adapter model is a much larger change. Deferred to a separate future ADR; this ADR extracts only the cleanly-decoupled tooling layer"* (`0017-surface-tooling-arc-bundles.md:64-67`). **This ADR is that future ADR.** It extends ADR-0017's arc-bundle shape from the tooling layer to the adapter layer.

The stocktake grounding this decision is in `docs/design-pluggable-adapters.md` §2 (verified @ v6.7.0). The adapter contract is already clean (`src/adapters/types.ts:183-264`: `start/stop`, `updateConfig?`, `attachInboundDispatch?`), and both construction paths already route through one injected factory (`src/gateway/gateway-adapters.ts:264-347`; the S9/cortex#1523 `wireSurfaceAdapters` extraction). The residual couplings are enumerated as blockers #5–#8/#13 in the design doc.

**Renderers hit the same wall, for the same reason (fold-in, 2026-07-11).** Adding a renderer today requires editing a closed `RendererKind` enum (`src/common/types/cortex-config.ts:1020-1025`), the `RendererSchema` discriminated union (`:1193`), **and** the `createRenderer` switch (`src/renderers/index.ts:32-49`) — three in-tree edits and a cortex release. `cli-tail` and `webhook-out` have sat in that enum since MIG-7.2d as unimplemented kinds that throw `UnimplementedRendererKindError` (`:22-30`): the enum advertised a plugin point the factory never delivered. That is the renderer-side twin of the adapter blocker.

Meanwhile the *runtime* half of renderer pluggability already exists and is unused: `SurfaceRouter.register()` returns an `{ unregister }` handle (`src/bus/surface-router.ts:262`, implemented `:456-464`) that the renderer boot loop **discards** (`src/cortex.ts:3257`), and per-renderer boot fail-isolation is already implemented (`src/cortex.ts:3243-3249,3254-3268`). Renderers are therefore *cheaper* to make pluggable than adapters, not harder — the adapter side still lacks per-instance detach entirely (`src/gateway/surface-gateway.ts:167,191` is all-or-nothing, blocker #13). The full renderer stocktake (R1–R13) is `docs/design-pluggable-adapters.md` §2.2.

## Decision

**Surfaces become pluggable surface plugins: per-plugin arc bundles a principal installs/upgrades/removes independently of cortex releases, discovered and loaded by the running daemon behind ONE versioned plugin SDK, compat-gated at load, fail-isolated per plugin, running in-process.** Five pinned decisions (D1–D4 as originally drafted, extended to renderers where they differ; D5 is the fold-in):

**D1 — Compat gate: SDK semver range, authoritative at the loader.**
Cortex exports a single-integer-major `SURFACE_SDK_VERSION`. A plugin declares an `sdkRange` (e.g. `^1`) in its default-exported `SurfacePlugin`. **The loader** (runtime, inside the daemon) is the sole authoritative gate: it checks `satisfies(SURFACE_SDK_VERSION, plugin.sdkRange)` at `await import()` time and refuses a non-satisfying plugin with a `system.error` event, leaving the daemon and all other plugins live. A `cortex: >=X <Y` version range MAY appear in `cortex-plugin.yaml` as **advisory** metadata (arc install may surface it), but it is **not** the load gate. *Rationale: the SDK version is the honest contract — it breaks only when the plugin API breaks — so it decouples surface compat from cortex's release cadence (the epic's core goal); a cortex-version gate re-couples them and goes stale on every unrelated cortex release. Only the running daemon knows its true `SURFACE_SDK_VERSION`, so arc install can only warn.* Cost accepted: `SURFACE_SDK_VERSION` is a disciplined semver contract — a breaking `PlatformAdapter`/`Renderer`/`SurfacePlugin` change is a *major* bump with a documented plugin-author migration.
> **Renderer delta: none.** One SDK barrel, one version constant, one gate, both kinds (D5). A breaking change to `Renderer` bumps the same major as a breaking change to `PlatformAdapter` — plugins of the *other* kind are then conservatively refused despite being unaffected. Accepted: coarse but honest, and the alternative (two independently-versioned SDKs) reintroduces the two-loaders problem D5 exists to prevent. Revisit only if renderer-contract churn measurably outpaces adapter-contract churn.

**D2 — Extraction deletes the in-tree implementation; never-extracted plugins are the permanent contract anchors.**
Each extraction PR (S9–S12, S12b) removes the real in-tree implementation *and* its npm deps once its bundle is proven. There is **no fallback copy** in cortex. *Rationale: a fallback is a second source of truth and, decisively, the platform deps (blocker #8) cannot leave `package.json` while any in-tree module imports them — keep-as-fallback would never achieve the dependency extraction that is half the point.*
> **The rule is "no plugin has two sources of truth" — not "cortex ships no in-tree plugins."** Two plugins never extract and stay in-tree permanently as the registry's contract anchors, exercising both construction paths in CI: the **mock adapter** (adapter side, a test double) and the **`dashboard` renderer** (renderer side, a real shipping renderer). **Renderer delta / exception (OQ8):** `dashboard` stays because removing `kind: dashboard` is a **config-breaking change** (`src/renderers/dashboard.ts:18-21`) and because it is one half of the **G-1111 §4.6 fail-safe pair** covering `local.{principal}.system.>` (`src/renderers/types.ts:15-21`). It does **NOT** stay "because it serves Mission Control" — verified false: `DashboardRenderer` is an inert ring-buffer stub, superseded as the MC feed by ADR-0005 §4, and nothing in production reads its `getRecent()` (`src/renderers/dashboard.ts:1-32`); MC's live feed is a *different* render target, `src/surface/mc/projection/dispatch-lifecycle-renderer.ts`. The exception is correct; the intuitive reason for it is wrong, and this ADR records that so a future reader does not "fix" the dashboard renderer into MC's critical path.

**D3 — v1 MVP is boot-time discovery; runtime load/unload/reload verbs are a separate slice (S8).**
v1 (S6) discovers, compat-gates, imports, validates, and registers plugins **at boot**; installing a bundle + restarting the daemon brings a surface live with no cortex release. Runtime `cortex plugin list|load|unload|reload` (S8) is a distinct, later slice carrying: module-cache busting (query-param import; the old module tree is dereferenced, not evicted — a **documented, bounded one-tree-per-reload leak**, so `unload` MUST `stop()` and drop every reference first), in-flight drain (inbound stops immediately; the outbound sink drains a bounded window then detaches — a mid-flight dispatch losing its sink is a `system.error`, never a silent drop), per-adapter attach/detach on gateway+router (blocker #13 — `surface-gateway.ts:167,191` is all-or-nothing today), and **state loss on reload by design** (adapter connection state + progress placeholders reset; nothing durable is lost — response routing is wire-level). *Rationale: boot-discovery delivers the epic's core value at low risk and fits the isolated trust-path review lane the wave plan gives S6; runtime verbs' complexity is sequenced after.*
> **Renderer delta: S8's renderer half is small, and it reverses a documented decision.** Per-instance detach for render targets **already exists** — `SurfaceRouter.register()` returns `{ unregister }` (`src/bus/surface-router.ts:262`, implemented `:456-464`); the boot loop merely drops the handle (`src/cortex.ts:3257`). So there is no attach/detach subsystem to build and **no drain machinery**: a renderer has no inbound to stop, and the router already bounds every `render()` with a timeout + `Promise.allSettled` isolation (`src/renderers/types.ts:36-39`), so `unregister()` halts new renders instantly and at most one in-flight render outstands. Reload is **detach → destroy → construct → attach**, never in-place mutation. State loss is by design and harmless (the dashboard ring buffer resets — nothing reads it; PagerDuty's `dedup_key` is derived per-envelope from `${envelope.source}:${envelope.type}`, `src/renderers/pagerduty.ts:23-24`, so it is reload-stable). **One new hazard:** reloading `pagerduty` detaches the paging sink for the reload window, breaking G-1111 §4.6 coverage transiently — S8 MUST **attach-before-detach** (register the new instance first; the router tolerates two targets on the same subjects, and PagerDuty's `dedup_key` absorbs the duplicate). Duplicate delivery is strictly safer than a blind window for a pager.

**D4 — Plugins run in-process with full daemon authority; risk accepted for v1 with mitigations.**
A plugin loaded into the stack daemon has the daemon's full authority — the stack NKey seed (it can sign as the stack), config secrets, CC-session spawn, bash, filesystem — and `await import()` executes arbitrary plugin code at load. **A malicious/compromised plugin = full stack compromise. The compat gate is not a security gate; fail-isolation isolates crashes/incompat, not malice.** Accepted for v1 because every plugin is first-party. Mitigations: (1) **org-trusted repos only** — v1 loads only `the-metafactory` bundles via arc's repo-first install (ADR-0017), no by-name/third-party loads; (2) the **same blocking CI floor as cortex** on bundle repos (`check-shippable-hygiene.ts` + `confidentiality-gate`); (3) **provenance pinning** — arc pins repo+ref and a bundle upgrade is a code-review event, not a passive fetch; (4) **"fail-isolation ≠ sandbox" is stated** so the isolation guarantee is never over-read. **Named future escalation:** registry signing (mirrors ADR-0018/0023), and separate-process-over-IPC if a real trust boundary is ever required (the gateway's separate-process unsigned/stack-re-signs model is the partial precedent).
> **Renderer delta: none — and explicitly no lower-risk carve-out.** A renderer bundle's top-level code runs at `import()` with the same authority as an adapter's. Renderer configs also carry **real secrets**: `PagerDutyRendererSchema.routingKey` (`src/common/types/cortex-config.ts:1142`) and `WebhookOutRendererSchema.authHeader` (`:1179`). What a renderer lacks is a **binding** (the agent↔surface credential mapping in `surfaces.*`), not secrets. All four mitigations apply to renderer bundles unchanged. *(This corrects a fold-in premise that renderers have "no secret fields"; there is no `secretFields` concept anywhere in cortex — `grep -rniE 'secret_?fields' src/ docs/` is empty.)*

**D5 — Two plugin classes, ONE SDK barrel and ONE loader. (fold-in, principal-approved 2026-07-11)**
A **platform adapter** and a **renderer** remain distinct concepts — `CONTEXT.md` is the definition and this ADR does not blur it. They are **not** merged into a single interface. But they are packaged, discovered, gated, isolated, and registered by **one** mechanism, because those are properties of *loading code into the daemon*, not of what the code does. Shared: one manifest (`cortex-plugin.yaml`, discriminated by `kind: adapter | renderer`); one discovery + `import()` + shape-validation path; one compat gate (D1); one per-plugin fail-isolation policy; one `SurfacePluginRegistry` keyed by `(kind, id)`; one trust posture (D4); one SDK barrel (`src/surface-sdk/`) exporting **both** `PlatformAdapter` and `Renderer` plus `SURFACE_SDK_VERSION`. Differs by kind: an adapter contributes a **binding schema** to `SurfacesSchema`, a **demux key**, and optional **`groupBindings`** (Discord token grouping); a renderer contributes a **config schema** to `RendererSchema`, a `rendererKind`, and `subscribe` subjects — and has no binding, no demux key, no token grouping. An adapter is a **dispatch source and sink**; a renderer is a **dispatch sink only** and never publishes as a side effect of rendering (`src/renderers/types.ts:40-43`). *Rationale: we refuse to build the loader twice. The alternative — a merged single class — would force a renderer to stub ten adapter-only methods and would destroy the type-level guarantee that a render target cannot publish.*
> Consequently `RendererKindSchema` (`src/common/types/cortex-config.ts:1020-1025`) stops being a closed `z.enum` and becomes the registered key set; `RendererSchema` (`:1193`) composes from registry-contributed `configSchema`s; and `UnimplementedRendererKindError` (`src/renderers/index.ts:22-30`) is **deleted** — an unknown kind is no longer "in the schema but unimplemented", it is simply *not registered*, which is the honest error.

**Load posture (recommendation, pending OQ6):** external-plugin loading is **default-off behind a `system.plugins.external` flag** (renamed from the wave plan's `system.adapters.external` — it gates renderers too, so the adapter-scoped name would be a lie) — in-tree registrations always load; external plugins load only when the principal opts a stack in. Matches the dev-loop-dormant-by-default secure posture. **⚠ Renderer tension (OQ9):** for the `pagerduty` renderer, "don't load external code by default" is **fail-open** — a stack with the bundle installed but the flag off has an installed pager that never pages, and its only remaining `local.{principal}.system.>` sink is the inert dashboard. A default that is *secure* for adapters is *unsafe* for a paging renderer. This must be resolved before `pagerduty` extracts (see Consequences).

## Consequences

- **Cortex shrinks and stops release-gating surfaces.** A surface change is a bundle release; a cortex release no longer touches surfaces (D1). Platform npm deps leave `package.json` as each adapter extracts (D2, blocker #8). *Renderers contribute no dependency-extraction win — neither in-tree renderer has an npm dep (`src/renderers/{dashboard,pagerduty}.ts` import types only). The renderer payoff is purely "add a sink without a cortex release" (D5).*
- **One new public contract to maintain with discipline: the plugin SDK.** `SURFACE_SDK_VERSION` + `src/surface-sdk/` become a versioned surface carrying **both** `PlatformAdapter` and `Renderer`; breaking either is a major bump + migration (D1, D5, OQ5).
- **The core is rewired once (S3–S4), then extraction is mechanical.** The fixed 4-method `GatewayAdapterFactory` becomes a `(kind, id)`-keyed `SurfacePluginRegistry`; `SurfacesSchema` composes registry-contributed binding schemas instead of 4 hardcoded `.strict()` keys, and `RendererSchema` composes registry-contributed config schemas instead of a fixed discriminated union. Because both construction paths already share one factory (S9), **the epic's blocker #4 is mostly retired** — S3 is "make the factory a registry," not "unify two sites." The real residual `cortex.ts` coupling is **blocker #6** (the worklog `DiscordAdapter.getClient()` reach-through at `:134,335,5889,5936`), which S2 must clear before the discord extraction (S12).
- **Discord extracts last, hard-gated on S1+S2.** Deleting `src/adapters/discord/` before the generic modules (`attachment-types`/`attachments`/`channel-context`, `dispatch-handler.ts:25,51,52`) relocate and the worklog decouples would break the core.
- **A new trust surface exists.** In-process plugin code runs with full authority (D4) — renderers included, and their configs carry secrets. The mitigations bound likelihood, not blast radius. Any future need for untrusted plugins forces the separate-process escalation.
- **⚠ REVERSAL: `src/renderers/types.ts:45-57` ("Renderers are static across hot-reload") no longer holds.** That comment is a decision record and is reversed here, on its own stated terms, rather than silently contradicted. Its rationale was: *"a renderer's resources … are scoped to its lifetime, and the failure modes of partial hot-reload (a routing key swap mid-flight) are easier to reason about with a full stop/start cycle … A future iteration MAY add an `updateConfig(RendererConfig)` hook if the principal-visible cost of the restart proves too high (Holly cycle 1 W3)."* Three premises; the fold-in moves two:
  1. **"A restart is cheap enough" — no longer true.** Under D5 a renderer arrives as a separately-installable bundle whose entire promise is "activate without a cortex release." Requiring a full daemon restart to activate a PagerDuty sink drags every live adapter connection and in-flight dispatch on that stack through a restart for an unrelated plugin. The comment itself named this trigger: the principal-visible cost of the restart has proven too high.
  2. **"Partial hot-reload is harder to reason about" — honoured, not contradicted.** We do **not** add `updateConfig(RendererConfig)`; the mid-flight routing-key swap the comment feared is never performed. We add **detach → destroy → construct → attach** via the `unregister()` handle that already exists (`src/bus/surface-router.ts:262,456-464`): the target leaves the router's dispatch list *before* teardown, so no render can observe a half-swapped renderer. Full stop/start remains the semantic; its blast radius shrinks from *the daemon* to *one renderer*.
  3. **"A renderer's resources are scoped to its lifetime" — still true, and now argues for us.** Owning nothing durable and never publishing is exactly what makes per-instance destroy-and-reconstruct safe for a renderer where it is not for an adapter (which drops a live platform connection and orphans progress placeholders).
  **Action:** the S8 renderer sub-slice rewrites that doc comment to cite ADR-0024. It must not be left contradicting the shipped behaviour.
- **Three fold-in premises were corrected against the code, and the corrections are load-bearing:** (a) renderers **do** carry secrets (`routingKey` `cortex-config.ts:1142`, `authHeader` `:1179`) — no reduced-trust carve-out (D4); (b) the `dashboard` renderer stays in-tree for **config-compat + G-1111 §4.6** reasons, **not** because it serves Mission Control (it does not — `src/renderers/dashboard.ts:1-32`) (D2); (c) `renderer.id` is a hardcoded per-kind constant (`dashboard.ts:59`, `pagerduty.ts:73`), so a `(kind, id)`-keyed *instance* registry collides on two same-kind `renderers[]` entries — S3 must add an optional `renderers[].id` defaulting to `kind` (OQ10). The registry keys plugin **classes**; instances are keyed separately.
- **Extracting `pagerduty` can silently disable a stack's paging sink.** It is one half of the G-1111 §4.6 fail-safe pair; after extraction, a stack that has not installed the bundle — or has installed it with `system.plugins.external` off, the *recommended secure default* — is left with one, inert, sink on `local.{principal}.system.>`. The coverage rule "fires at config-load, not in the Zod schema" (`src/common/types/cortex-config.ts:1015-1018`) and must learn to distinguish *misconfigured* from *bundle-not-loaded*. **`pagerduty` extraction (S12b) is HELD until OQ9 is settled.**
- **Standing cross-reference:** this ADR is the answer to ADR-0017's deferred "extract the adapter too" alternative. Future reviews cite this ADR rather than re-opening 0017.
- **This ADR is the ground truth once accepted.** A downstream issue (S1–S13) that disagrees updates itself first (comment the delta), then implements — never silently diverges (epic working notes).

## Alternatives considered

- **Separate-process adapters over IPC (surfaces as sandboxed out-of-process plugins).** Each adapter runs in its own process, talking to the daemon over IPC/a local socket; a compromised adapter is confined to its process. **Rejected for v1** on cost and latency: it adds a process-supervision layer, an IPC serialization boundary on the hot inbound/outbound path, and per-adapter lifecycle management — disproportionate while every bundle is first-party and the accepted-risk mitigations (D4) hold. **Explicitly retained as the named escalation path** if a real trust boundary is ever required; the surface gateway (cortex#524) already runs adapters out-of-process and unsigned (`CONTEXT.md:181`), and ADR-0012 is the account-isolation precedent — so the door is architecturally open, not closed.
- **Keep adapters in-tree behind build flags / conditional compilation.** A build/env flag includes or excludes each adapter at bundle time. **Rejected:** it does not remove the source or the npm deps from the repo (blocker #8 unresolved — `discord.js` still pinned into every build's dependency graph), still requires a cortex release to change the flag set, and gives no independent per-surface versioning. It is monolith-with-a-switch, not pluggability.
- **Status quo — adapters stay compiled into core.** **Rejected:** it is the problem statement — monolith growth, whole-cortex releases for any surface change, every stack carrying every surface's deps, and it leaves ADR-0017's deferral permanently open.
- **cortex-version-range as the primary compat gate** (bundle declares `cortex: >=X <Y`). **Rejected as the gate** (kept as advisory metadata, D1): it couples surface compat to cortex's release cadence and goes stale on every cortex release that doesn't touch the adapter contract, and arc-install-time checks mislead after `arc upgrade cortex`. The SDK semver range tracks the thing that actually breaks compatibility.
- **Keep an in-tree real-adapter fallback after extraction.** **Rejected** (D2): a second source of truth, and it blocks the dependency extraction (blocker #8) for as long as it lives. A time-boxed transitional fallback is the conservative variant offered to the principal (design §7 OQ2), but the recommendation is delete-on-extraction with the mock as anchor.
- **One merged plugin class (a renderer is an adapter that stubs the inbound half).** A single `SurfacePlugin` *interface*, with renderers no-op-implementing the ten adapter-only required methods (`getPlatformUserId`, `fetchContext`, `resolveAccess`, `postResponse`, `sendTyping`, `sendProgress`, `clearProgress`, `createThread`, `resolveLogicalTarget`, `notifyPrincipal` — `src/adapters/types.ts:205-249`). **Rejected** (D5): it buys nothing the shared *loader* does not already buy, and it costs the property that makes the surface-router safe — a render target structurally **cannot** publish or hold a chat credential, and the type system should enforce that rather than rely on ten well-behaved stubs. It would also contradict `CONTEXT.md`, which defines adapter and renderer as distinct concepts separated by three properties (inbound / agent-bound / lifecycle). The shared thing is *loading code into the daemon*, not the contract.
- **Adapters only; renderers stay in-tree behind their closed enum.** **Rejected** (D5): renderers hit the identical structural blocker — `RendererKind` is a closed `z.enum` (`src/common/types/cortex-config.ts:1020-1025`) and `createRenderer` a closed switch (`src/renderers/index.ts:32-49`), so adding one is three in-tree edits and a cortex release; `cli-tail`/`webhook-out` have sat there unimplemented, throwing (`:22-30`), since MIG-7.2d. Deferring them means either (a) building the loader, registry, manifest, compat gate, and fail-isolation a **second time** later for renderers, or (b) never making renderers pluggable at all. The marginal cost of admitting a second `kind` to a loader that must exist anyway is one discriminant field; the marginal cost of a second loader is the whole slice again. Renderers are additionally the *cheaper* half — per-instance detach already exists (`src/bus/surface-router.ts:262`) and boot fail-isolation already exists (`src/cortex.ts:3254-3268`).

### OQ9 — RATIFIED: fail loud, never page into the void

**Decision: (b) + (a).** A stack whose `local.{principal}.system.>` coverage drops below **two distinct platform classes** **hard-fails boot**, naming the missing bundle and the `arc install` that fixes it. **First-party renderer bundles are exempt from the default-off `system.plugins.external` flag**, so an installed first-party pager loads without the principal opting in.

**Rationale.** A default that is *secure* for adapters ("don't load third-party code into the daemon") is **fail-open** for a pager ("don't page"). Between the two failure modes — a stack that refuses to start, and a stack that runs blind while believing it is monitored — the loud one is correct for an alerting sink. The risk is deliberately moved from *silent no-page* to *loud no-boot*.

**Implementation constraints this imposes:**
- The G-1111 §4.6 coverage check fires at **config-load** (`src/common/types/cortex-config.ts`), not in the Zod schema. It must learn to distinguish *"you configured one sink"* (a **config** error) from *"you configured two, one's bundle isn't loaded"* (an **install-state** error). Two distinct messages; the second names the bundle and the remedy.
- "First-party" must be a checkable property of a bundle, not a naming convention. Define it where the loader can evaluate it (S6), and record it — an unchecked "first-party" exemption is a trust hole wearing a friendly name.
- `DashboardRenderer` is **inert** (ADR-0005 §4): it counts toward the *class* pair but delivers nothing. Coverage counting must not be fooled into treating "dashboard alone" as adequate — that is the exact fail-open this decision closes.

**Consequence for S12b:** the pagerduty renderer extraction is **unblocked**, but MUST land the boot-coverage hard-fail *before or with* the extraction, never after.
